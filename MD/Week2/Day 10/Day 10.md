## 객체지향
* 정적인 관계 - 객체의 역할과 포함 관계, 순간적인 구조 설명 가능
	* 프로그램 실행 없이도 코드 자체만 봐도 알 수 있음
	* 상속 / 연관 / 포함 => 관계의 수명이 길고 객체의 상태 변화와 상관 없이 유지
* 동적인 관계 : 사용 관계로 객체 간 종속성을 알려주는 관계
	* 실행되는 도중 맺어지거나 변하는 관계
	* 의존 / 다형성 => 관계 일시적, 객체의 상호작용과 흐름에 집중
* 사용 관계는 화살표로 표시
	* 화살표 시작 객체가 닿아 있는 객체 사용
	* 사용 객체가 사용되는 객체에 종속적
	* Diagram으로 활용 => UML(User Modeling Language)
		* 클래스 다이어그램(class diagram) -> 클래스의 속성, 메서드, 상속/포함 관계, 정적
		* 객체 다이어그램(object diagram) -> 특정 시점의 객체 인스턴스 상태, 정적
		* 유스케이스 다이어그램(Use Case Diagram) -> 사용자가 시스템을 통해 무엇을 할 수 있는지, 동적
		* 시퀀스 다이어그램(Sequence diagram) -> 객체들이 어떤 순서로 메시지를 주고 받는지, 동적
		* 상태 다이어그램(State Diagram) : 객체의 상태 변화 추적, 동적
* 대표 객체와 파생 객체
	* 비슷한 역할을 하는 객체의 공통 특성만 포함한 대표 객체
	* 대표 객체의 공통 특성을 물려받고 새로운 특성을 추가한 한 파생 객체
		* 상속 : 대표(상위) 객체에서 파생 객체를 만드는 것
	* 대표 객체의 특성을 상속 받아 상세화 한 파생 객체
	* is-a 관계는 상호 교환 가능(has-a 관계는 불가)
	* Super(Base, Parent) set <-> Sub(Penved, Child) set
	* Is-a : 상속, Has-a : 소유, Uses-a : 사용
* 여러 계층의 중첩 표현
	* 추상화를 여러 계층으로 진행, 복잡한 논리 구조를 표현
	* 동력 장치 모델링을 엔진 하위에 내연 기관 엔진과 전기 엔진으로 분류
		* 총 2단계 구조에서 3단계 구조로 세분화
		* downcasting 할 때 유의 -> 선조가 같더라도 형제끼리의 연관성은 없음
* 객체 간에 접근 제어
	* 직접 사용하지 않는 이벤트 사용 관계
		* 객체의 동작을 직접 제어하지 않고 특정 이벤트로 제어하는 방법
	* 접근이 금지된 객체에 접근할 수 없어야 함(접근 제어)

#### 캡슐화
* 복잡한 내부 기능 묶어 단순화해 동작, 외부에서 불필요한 정보 감추기
	* 함수(클래스)라는 커다란 덩어리로 묶어 필요한 부분만 노출
#### 추상화
* 객체의 공통적인 속성과 기능 추출, 본질적인 것만 정의하고 나머지 생략
	* 구체적인 구현보다는 무엇을 하는가에 집중
#### 상속
* 부모 객체의 특성을 이어 받아 새로운 객체를 만드는 것
* 상속받은 특성 이외에 상속 받은 객체에 고유의 특성을 추가하는 것이 필요
* 후손 클래스 객체의 크기가 더 큼(선조 + a)
#### 다형성
* 부모 객체의 특성을 모두 갖고 있는 자식 객체는 부모 객체 대체 가능
* 부모 객체를 대체한 자식은 자신만의 특성 반영, 자식 객체마다 다르게 동작

## 클래스와 인스턴스
#### 클래스
* 데이터와 함수를 포함하는 논리 단위 -> 객체 = 클래스
	* 객체와 클래스는 변수와 데이터 형식과 유사한 관계
* 클래스 - 객체가 포함하는 데이터와 함수를 정의하는 문법적인 요소
	* 데이터와 함수를 클래스 정의에 포함, 멤버 변수(데이터), 멤버 함수(함수)로 호칭
	* 멤버 변수, 멤버 함수는 접근 통제를 위해 public, private 같은 키워드 사용
	* 생성자와 소멸자 - 객체 생성 될 때와 소멸 될 때 자동으로 호출되는 특수 함수
* 객체 - 특정 클래스를 사용할 수 있도록 만든 변수
* 클래스 내부 함수를 클래스 외부에서 선언 시 datatype class_name :: func_name() {}형으로 선언
	* :: -> 범위 지정 연산자임
* 클래스 선언부 : 멤버 함수 정의를 제외, 클래스와 관련된 모든 것 선언
* 멤버 함수 정의부 : 클래스에 선언한 멤버 함수 정의(내부)
* 부모 클래스(선택) : 상속 받을 때 콜론 다음에 접근 지정자와 부모 클래스 지정
* 접근 지정자 : 접근 여부 표시, 이후 따르는 멤버는 접근 제어 설정에 따름
* 생성자/소멸자(선택) : 객체 생성/소멸 시 호출, 선택적 사용
* 앵간해서 body는 따로 소스파일 만들어서 정의하는게 좋음
#### 객체 선언
* 정의된 클래스를 사용하기 위해서는 객체 선언이 필요
* 데이터 선언과 유사한 정적 객체 선언과 new를 사용한 동적 객체 선언
	* engine \*my_engine_pointer = new engine();
	* delete my_engine_pointer;
* 정적 클래스를 사용할 수 있도록 만든 변수
	* my_engine.current_fuel;
	* My_engine.increasing_piston_speed();
* UML에서 선조는 화살표 헤드쪽임
* 평서체 : body를 갖고 있음
	* 소스 코드를 만들어줘야 함(func의 body)
* 이탈릭체 : body를 갖고 있지 않음
	* 구현부가 없는 추상 함수, 자식 클래스에서 overriding
* -로 기재되어있으면 private, + public, # protected, ~ package
* 객체 선언 시 해당 객체가 부모 클래스에서 상속 받은 객체라면 스킵해도 됨, 오히려 선언 시 가려지는 문제 발생 가능
#### 추상화
* 공통 특징은 추출, 불필요한 부분 제거를 통한 간결화 및 이해 쉽게 만드는 작업
* 과정
	* 현실 세계 대상 추상화, 프로그래밍 언어 모델로 변경하는 과정
	* 공통 특성 추출, 불필요한 요소 제거해 멤버 변수, 멤버 함수 규정
* 물리적 특성과 행위적 특성을 나눔(클래스는 이 두 개 모두 필요)
	* 절차는 함수가 우선권, 클래스는 객체가 우선권 => 무조건 물리적 특성부터 뽑기
#### 캡슐화
* 변수와 함수를 클래스로 감싸 외부에서 직접 접근 할 수 없도록 은닉화
* 하나의 기능이 여러 함수로 이뤄질 경우 대표 함수만 노출
	* 클래스 내부의 세부 로직을 알 필요 없음 -> 클래스 변경이 외부 로직에 영향 안 줌
* 접근 지정자
	* public(+) - 타 클래스 어디서든 접근
	* private(-) - 같은 클래스 안의 멤버 함수에서만 접근
	* protect(#) - 클래스 내 멤버 변수나 상속 관계의 클래스에서 접근
	* cpp에서는 구조체도 멤버 함수 포함 가능, 기본 접근 지정자는 클래스와 다르게 public 선언
* getter / setter
	* 값이 변경되지 않도록 보안성 증가
	* 값의 유효성 검증을 위해 사용
	* getter를 안만들고 setter만 만들면 write only 변수
	* setter를 만들고 getter만 사용하면 read only 변수
#### 캡슐화
* 클래스, 객체, 인스턴스
	* main함수에서 첫 줄에 있는 bank my_bank는 인스턴스 생성 코드
	* 인스턴스는 클래스가 메모리에 할당되어 생성된 변수
* 객체
	* 논리적 개념으로 클래스로 구현하고자 하는 구현 대상, 또는 인스턴스를 달리 부를 때 사용
* 클래스
	* 객체를 구현하기 위한 설계도
* 인스턴스
	* 클래스 정의에 따라 메모리에 실체화된 객체의 형태
* bank my_bank; -> bank 클래스를 사용할 수 있는 인스턴스를 my_bank라는 이름으로 선언
	* my_bank라는 이름으로 bank 클래스의 멤버 함수인 use_counter 호출 가능
#### Message / Method
* Message : function call -> 함수 호출
	* 메시지를 전달하면
* Method : function run -> 함수 실행
	* 메소드 실행
* this라는 보이지 않는 포인터를 통해 메시지를 보내 메소드를 실행(this->)
	* 멤버 함수 호출 시 호출 객체의 주소를 담고 있음
	* 동일한 함수를 쓰는 객체 중 어떤 데이터를 건드릴지 결정해주는 유일한 단서
	* 멤버 함수는 공유되지만, 데이터는 각자 지님. 해당 데이터를 연결해주는 것이 this
#### 추상화와 캡슐화를 이용한 클래스 설계
* 예시 : 몬스터 설계, 아래 규칙으로 움직임
1. 몬스터는 체력과 공격력이 있다
2. 몬스터는 데미지를 입으면 체력을 소모한다
3. 몬스터는 공격할 수 있다
4. 몬스터는 특수 공격을 할 수 있다
5. 몬스터의 종류에 따라 공격력과 특수 공격은 다양하다
