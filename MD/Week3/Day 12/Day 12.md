## 생성자와 소멸자
#### 복사 생성자
* 동일한 클래스 기반 객체 2개 생성
* 다수 객체 생성
	* 초기화 코드가 있는 멤버 변수의 값을 변경하려면 함수 호출 또는 값 대입 필요
	* 설정 값이 조금씩 다른 몇 십 개의 객체 생성이 매우 번잡스럽게 작업
* 복사 생성자
	* 초기화가 아닌 객체를 복사할 때의 값들을 복사해서 이미 설정된 값을 사용해 객체를 새롭게 생성
	* 동일한 설정 값은 추가적인 설정이 필요 없음, 차이나는 멤버 변수 값만 설정
	* 포인터를 이용, call by reference 형태로 사용
		* 값 전달 방식을 쓰면 인자를 넘길 때 또 복사 생성자 호출, 무한 재귀
		* 이를 방지 및 메모리 낭비 줄이기 위해 참조 사용
		* const 사용 이유 : 복자 대상 원복 객체 수정 방지
	* 객체와 객체간의 멤버 변수 복사 -> bitwise copy -> 주소 다시 복사
		* 포인터 변수가 존재, 같은 주소를 가리킴(side copy, shallow copy)
		* 따라서 주소만 복제하는 것이 아닌, 구조까지 복사해야함(deep copy)
	* 기본 복사 생성자는 bitwise copy 동작
* delete
	* 지우는 요소의 정보 기재해줘야함
	* 배열을 지울땐 배열 delete(delete\[])를 사용
#### 소멸자
* 생성자와 반대
	* 객체가 소멸할 때 필요한 메모리 해제나 환경을 이전 상태로 변경하는 등의 일를 진행
	* 생성자와 비슷하게 소멸될 때 자동으로 호출
* 기본 소멸자
	* 이름은 클래스 이름과 같게, 이름 앞에 물결표 추가
	* 기본 소멸자 이외에 소멸자 만들 수 없음
	* 다중 상속인 경우 소멸자는 생성자 호출의 역순으로 호출
* 소멸자는 기본적으로 가상 소멸자인 경우 정상 동작

## this 포인터
#### 객체가 메모리에 할당되는 구조
* 객체도 하나의 변수, 클래스 크기만큼 메모리 할당
	* 클래스를 표현하기 위한 메모리, 멤버 변수를 저장하기 위한 메모리 등
* 멤버 함수는 동일 클래스의 모든 객체가 공통 사용
	* 멤버 함수는 메모리에 한 번만 적재, 같은 클래스의 객체를 추가하더라도 똑같은 함수 참조
=> 자기 자신을 가리킨다는 표현 보다는 호출이 일어날 때 메시지를 받은 객체를 가리킴
#### this 포인터
* 멤버 변수와 멤버 함수는 메모리 위치가 다름
	* 원칙적으로 멤버 함수에서 멤버 변수의 값을 변경 불가(주소 모름)
	* 프로그램 정상 동작
* 객체의 메모리 주소(멤버 변수 등)를 참조하는 this 포인터
	* this 포인터는 멤버 함수 내에서 자동 생성, 해당 함수가 속한 객체의 주소 참조
* 함수 체이닝
	* 여러 함수 연이어 호출
	* 멤버 함수가 객체 반환, 해당 객체에서 다시 멤버 함수 호출
	* 코드 간결하게 만들고 호출 순서를 직관적으로 표현
	* 객체에 메시지를 전달한다고 생각하면 직관적인 이해 가능
#### 함수 오버로딩
* 오버라이딩 : 자식 클래스에 정의된 함수 우선시
	* 다형성 동작을 위해선 dynamic binding 필수
	* overriding + virtual -> 다형성
* 오버로딩 : 정의된 함수와 같은 이름 사용, 매개변수 구성 변경
	* 새로운 정의를 계속 쌓는 의미
#### 연산자 오버로딩
* operator 키워드를 오버로딩, 연산자 재정의 가능

## 접근 지정자와 프렌드
#### 접근 지정자
* 멤버 변수를 정의하거나 상속받을 때 주로 사용
* 클래스 멤버 선언부에서 public, private, protected : 
	* 해당 접근 지정자로 정의할 멤버 작성 -> 외부 접근 제어
* 클래스 외부에서 클래스 멤버 함수.변수 사용 제어
	* 자식 클래스도 부모 클래스의 private 멤버에는 접근 불가
#### 상속에 따른 접근 지정자의 변화
* 상속 접근 지정자
	* 상속 받은 멤버의 접근 지정자 변경
	* 상속 접근 지정자에 따라 다르게 변경
		* public -> pub, pri, pro
		* protected -> pro, pro, pri
		* private -> pri, pri, pri
* 상속 접근 지정자 생략
	* 기본 접근 지정자 적용
	* cpp에서 기본 접근 지정자는 private, 구조체는 public
#### friend class
* 외부에서 접근이 불가한 private, protected 멤버의 접근을 예외적으로 허용
* 클래스 전체가 아닌 특정 함수에만 접근을 예외적으로 허용

## 객체지향을 돕는 기능들
#### 컴포지션과 어그리게이션
* 다중 상속 -> 부모 클래스를 여러 개 상속, 자식 클래스 정의
	* 다양한 부모 클래스 상속, 많은 부분 이미 정의
	* 부모 클래스에 정의된 안정적이고 견고한 디자인 패턴 그대로 활용
		* 개발 방법론이나 구조를 흔들림 없이 빠르게 전파 할 수 있음
	* 단점
		* 클래스가 커짐
			* 클래스가 다양한 역할을 수행하게 되는 거대 클래스 지양
				* 속성과 기능 많아 사용 어려움, 부모 클래스가 변경되면 상속받은 모든 클래스에 영향
				* 변경 사항이 여러 곳에 영향을 줌, 바람직하지 않음
			* 상속이나 사용 관계로 의존도가 높아지면 결합도가 높아짐
				* 결합도가 낮을수록 유지 보수 수월
		* 컴파일 시간 증가
			* 소스 코드가 변경되면 컴파일 재실행
				* 클래스 파일이 많으면 시간 증대
			* 변경된 부분만 재 컴파일
				* 결합도가 높으면 적은 수정에도 많은 파일 컴파일 진행
* 컴포지션과 어그리게이션
	* 멤버 변수로 포함해 클래스 재사용 -> 코드 작접 사용하지 않아 변경 시 재 컴파일 없음
	* 클래스는 단일 속성과 기능 가짐, 결합도 낮아지고 변경에 따른 영향 감소
	* 컴포지션은 포함, 어그리게이션은 사용 개념
* 컴포지션은 상속이 아니라 멤버 변수로 포함
	* 재사용할 속성과 기능을 별도로 분리, 해당 객체를 멤버 변수로 포함, 포함된 클래스에 종속
	* 생명 주기를 포함한 클래스가 관리, 두 클래스의 생명 주기 동일
* 어그리게이션
	* 부분 객체를 클래스 내부에서 직접 생성하지 않고, 외부에서 생성도나 객체의 주소를 인자로 전달받아 사용
	* 전체 객체가 소멸되어도 부분 객체는 사라지지 않아 계속 사용 가능
	* 외부 생성, 내부 사용
* 가상함수
	* 멤버 함수 가운데 자식 클래스에서 오버라이딩 해야 하는 함수
	* 일반 멤버 함수도 자식 클래스에서 오버라이딩 불가
		* 오버라이딩이 필요하다는 의미 전달 불가 -> 문법 모호
		* 부모 클래스로 업캐스팅 시 호출되는 함수 다름 -> 다형성 구현 원리
	* virtual은 선조, 후손 모두 다 추가
	* 오버라이딩 하는 자식 클래스에서는 override 키워드 추가
		* 선언부 가장 마지막 부분
## 가상 함수와 동적 바인딩
#### 함수의 동작 바인딩
* 가상 함수 호출의 원리
	* 가상 함수 테이블을 활용, 동적으로 바인딩
* 바인딩
	* 함수 호출이나 변수 참조가 동작 코드와 연결되는 과정
		* 프로그램 실행은 메모리를 이동하며 실행, 실행 코드 메모리를 연결하는 과정
	* 바인딩은 정적, 동적 두 종류
		* 각각 이른, 늦은 바인딩
	* 정적 바인딩은 컴파일 시점에 동작
		* 정적으로 바인딩 되는 대상은 컴파일 할 때 결정, 프로그램 실행 동안 유지
* 동적 바인딩
	* 대상이 실행 시점에 결정, 변경 가능
	* 가상 함수, 자식 클래스로 치환된 부모 클래스의 포인터가 대표 예시
	* 동적 바인딩 대상은 프로그램 실행 동안 수시로 변경 가능
	* 정적 : 멤버 함수 직접 호출
	* 동적 : 클래스 객체 이동 후 본 클래스 함수의 주소로 이동
#### 가상 함수 테이블
* 클래스 계층 구조 최상위에 존재하는 가상 함수에 대한 메모리 정보
* 업캐스팅 후에도 자식 테이블이 호출될 수 있도록 정보 제공하는 구조
* 가상 함수가 있는 객체를 생성하면 가상 함수 테이블을 가리키는 포인터 생성
	* 가상함수 호출 시 \_vfptr 활용, 가상 함수의 개수에 따라 크기 변경
#### 순수 가상 함수
* 순수 가상 함수란? -> body가 없음
	* 부모 클래스에서는 가상 함수를 정의 없이 선언만 진행
		* 강제적으로 후손 클래스에서 body를 만들어야함
	* 자식 클래스에서 반드시 재정의 해야 한다는 것을 문법적으로 알림
	* 순수 가상 함수로 명시적으로 남겨 둘 수 있음(순수 가상 함수로 다시 선언)
* 애매 모호한 추상 클래스, 객체 생성 금지 클래스
* 순수 가상 함수를 가지면 body가 없음 -> body를 만들어도 없다 취급
	* 문법적 선언으로, 컴파일러 단에서 함수가 논리적으로 미완 상태임을 명시해 객체 생성 불가를 선언함

#### 가상 소멸자
* 업캐스팅 된 객체의 소멸 시 문제 발생
	* 소멸자는 메모리 해제 등 리소스 정리를 위해서 사용
	* 업캐스팅 된 상태의 객체의 메모리가 해제 될 때 소멸자의 호출은
		* 업캐스팅 되지 않은 객체는 객체에서 정리한 소멸자, 부모 클래스의 소멸자 순으로 호출
	* 업캐스팅 된 경우 업캐스팅 된 클래스의 범위에서 소멸자 호출
		* 업캐스팅된 객체의 소멸자는 호출 안 됨
#### 추상 클래스
* 구체화되지 않은 추상적인 클래스
	* 순수 가상 함수를 포함하고 있어 구체화 되지 않음
	* 일반 멤버 변수나 멤버 함수를 포함 여부는 상관 없음
	* 다형성 구현 및 다양한 디자인 패턴에 활용
* 구상 클래스
	* 추상 클래스와 대비, 구체화 된 클래스
* 객체 선언이 불가능
	* 구체화되어 있지 않아 객체 선언 불가
	* 추상 클래스 상속, 순수 가상 함수를 오버라이팅한 클래스로 객체 선언
* 전략 패턴
	* 동일한 흐름에 다른 알고리즘을 사용하고 싶을 때 흐름과 알고리즘 분리
		* 흐름을 사용하는 곳에서는 어떻게 구현되었는지 신경쓰지 않아도 됨
	* 다른 알고리즘으로 쉽게 교체 가능
		* SW의 유지 보수성을 높히고 새로운 알고리즘을 추가하기 쉬움
* 인터페이스 => 추상화 클래스
	* 순수 가상 함수지만, 멤버 변수 없음, 함수로만 구성
	* 함수 뒤에 =0 선언 안해도 모두 순수 가상 함수로 선언
	* \_\_interface 사용 가능
=> 추상 클래스는 멤버 변수 가질 수 있음, body를 갖는 함수도 가능
=> interface는 오직 순수 가상 함수만
#### 정적 멤버
* 정적 멤버 변수, 정적 멤버 함수
	* static 키워드 사용, 객체 선언 없이 바로 사용 가능
	* 종속되지 않음
* 클래스당 한 번만 메모리에 할당
* 해당 클래스로 선언한 모든 인스턴스가 공유
* 소속 클래스를 범위 지정자 붙여 사용
* 클래스 안에서 선언, 초기화는 밖에서
* 정적 멤버 함수는 클래스 안에 정의해도 메모리에는 별도 존재, 인스턴스에 독립적

#### 정적 멤버 활용
* 정적 멤버 변수
	* 범위 지정자가 적용된 정적 변수, 같은 클래스에서 만들어진 객체에서 공유하는 값
	* 외부에서 참고할 수 있는 동일 클래스로 선언된 객체들의 값
* 정적 멤버 함수
	* 클래스의 객체를 클래스 내부에서 직접 관리하고 싶을 때, 또는 유틸리티 함수를 만들 때 활용
	* 유틸리티 함수 : 문자열 조작, 날짜, 시간 계산, 수학적 계산 등 특정 작업 지원 함수
	* 객체를 클래스 내부에서 직접 관리하는 예시 -> 팩토리 패턴
		* 객체 생성 클래스 멤버를 정적 멤버 함수로 정의

## 객체지향 설계 원칙
#### SOLID 원칙
* Single Responsibility Principle(SRP) : 단일 책임 원칙
* Open-Closed Principle(OCP) : 개방, 폐쇄 원칙
* Liskov Substitution Principle(LSP) : 리스코프 치환 원칙
* Interface Segregation Principle(ISP) : 인터페이스 분리 원칙
* Dependency Incersion Principle(DIP) : 의존성 역전 원칙
#### 단일 책임 원칙
* 클래스는 한 가지 기능만 수행, 한 가지 이유만으로 변경
* 산탄총 수술
	* 한 가지 기능을 수정할 때 클래스를 여러 개 수정해야 한다면 유지보수성 급락
	* 이러한 현상을 산탄총 수술이라 명칭, 탄흔을 사방에 남김
* 클래스 추출
	* 단일 책임 원칙을 적용하는 구체적인 방법 : 리펙터링
	* 상속 관계보다 컴포지션, 어그리게이션 적극 활용
	* 거대 클래스를 작은 단위로 분할
	* 분할된 클래스를 포함, 이전 논리적인 관계 유지
	* 기존 거대 클래스는 여러가지 기능을 묶는 역할
	* 추출한 하위 클래스를 has-a 관계로 설정
#### 개방 폐쇄 원칙
* 확장에 열려있고 수정에 닫혀있어야 한다
* 쉽게 확장(수정) 가능, main에서는 변화된 내용 몰라도 새로운 형태 동작 추가(폐쇄)
	* 동적 바인딩이 개방 폐쇄 원칙 설명
		* 새로운 기능 언제든 추가
		* 다른 코드에 파급 효과 없어 추가되는 기능 외에는 수정 필요 없음
* 추상 클래스 활용
	* 개발 폐쇄 원칙은 추상 클래스(인터페이스)로 구현 가능
		* 가능의 흐름은 추상 클래스 활용해 작성, 상속받아 구현하는 클래스에 세부 동작 구현
		* 흐름의 뼈대는 템플릿으로 만들고 살을 붙이는 작업은 자식 클래스에 위임
		* 템플릿 메서드 패턴
#### 리스코프 치환 원칙
* 하위 클래스는 상위 클래스를 대체할 수 있어야 한다
	* 다형성의 동작 원리 설명
* 자식 클래스가 부모 클래스 치환
	* 부모 클래스의 역할을 자식 클래스가 수행
	* 자식 클래스가 부모 클래스를 완전히 대체 가능한 is-a 관계
#### 인터페이스 분리 원칙
* 인터페이스는 작고 섬세, 클래스는 필요한 인터페이스만 구현
* 단일 책임 원칙과 연관
	* 클래스가 여러 인터페이스 상속 -> SRP 위반
	* 인터페이스가 여러 목적을 정의 -> SRP 위반
* 인터페이스는 반드시 최소의 기능만 정의
* 클래스는 최소한의 인터페이스만을 상속받아 구현
* 여러 기능이 필요한 클래스라면 has-a로 직접 구현이 아닌 멤버로 포함
#### 의존성 역전 원칙
* 상위 수준 모듈은 하위 수준의 모듈에 의존해서는 안되며
* 상위 하위 수준 모두 추상 레이어(인터페이스)에 의존해야한다
	* 의존성 -> 클래스가 타 클래스의 내용을 참조해 사용
* CPP에서는 가상 함수로 사용
