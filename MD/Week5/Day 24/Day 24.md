## 임베디드 기초
#### 임베디드 구조
* 임베디드 시스템은 반드시 대상(target)이 존재해야 함
	* 실제 디바이스(실물 장비)
* MCU -> Device(Control) -> Machine
	* 다양한 I/O를 통한 기계장치 제어
* 칩(Chip)에 대한 이해 필수 -> ASM(어셈블리)에 대한 기초 지식 필요
	* HW(하드웨어) 특성도 알아야 함
	* 각 레지스터의 의미(활용 방안) -> 단계별로 가능한 동작들

### 1장. 임베디드 시스템
* 현실 세계에서 사용하는 대부분의 형태는 실제 장비와 결합된 형태
	* 무엇을 할 건지(목적)에 따라 프로그래밍을 진행
		* 상위 레벨 언어를 통해 작성, 수정, 교체 가능
	* 범용적임 -> 여러 목적에 사용
* 임베디드 시스템은 위와 같은 작업(작성/수정/교체)이 쉽지 않음
	* 특정한 기계나 장치를 대상으로 함, 제어 목적을 가짐
	* 특정한 작업을 수행하기 위해 작성 및 제작됨
	* 동작 중에 기능이 변화하는 일이 거의 없음
#### NTCR(Nature, Time, Constraint, Reliability) -> 자연법칙, 실시간성, 제약조건, 신뢰성
* Nature : 자연 법칙의 취급
	* 외부 세계의 변화를 감지하고 MCU에서 처리한 뒤 결과를 외부 세계로 피드백하는 IPO
	* 예: 인체 감지 센서/거리 센서 등으로 사람을 감지해 자동문을 연다(센서-액추에이터)
	* Input: 변화 -> 칩에 전달되어 연산(Processing) -> Output: 행동(결과)
	* 입력과 결과 피드백 방식을 정의하는 제어 모델을 만드는 것이 중요
* Time : 실시간성의 요구
	* 현실에서 사용하는 대부분의 기기는 피드백이 즉시 나와야 하는 형태가 많음
	* 실시간성이 엄격하게 요구됨 -> Hard Real-Time -> 지연 허용이 거의 없음
	* 일정 수준의 지연을 허용 -> Soft Real-Time -> 적정 수준의 지연은 용인
![[Pasted image 20260423094426.png]]
	
	* Rising Edge / Falling Edge 기준으로 신호(signal)를 판단할 수 있다
	* Hard Real-Time -> 타이머 기준에 맞춰 즉시 처리됨
		* 반드시 정해진 deadline 안에 처리/응답해야 함
	* Soft Real-Time -> deadline을 놓쳐도 치명적 실패보다는 품질 저하 수준으로 수용되는 경우가 많음

* Constraint : 엄격한 제약 사항
	* 임베디드 시스템은 전용 기기로 제공되며, 여러 조건이 정의돼 있음
	* 해당 정의에 의해 임베디드 시스템에는 엄격한 제약사항 부여
	* 각 기기의 크기/무게, 가격, 이용 조건
	* 소비 전력이나 발열 문제에 대한 검토
	* 진동이나 먼지 등 가혹한 이용 환경 등
* Reliability : 높은 신뢰성
	* 어떤 제품/시스템이 주어진 조건에서 규정된 시간 내에 요구된 기능을 완료해 낼 수 있는 성질
	* 정해진 기간 동안의 고장률/오류 발생 빈도(예: MTBF 등)로 신뢰성을 표현
* NTCR을 기준으로 명세서 작성 필수

#### 하드웨어의 종류
| MPU(CPU)     | ALU, CU, R, Cache          | 연산 및 명령어 실행        |
| ------------ | -------------------------- | ------------------ |
| MCU          | MPU + RAM + ROM + I/O      | 임베디드 전용 SoC        |
| Memory       | RAM, ROM, Cache            | 데이터 저장(휘발성/비휘발성)   |
| Input device | 센서, 버튼, 키보드 등              | 외부 정보를 시스템으로 전달    |
| 출력 장치        | 모터, 액정, LED 등              | 처리 결과를 외부로 전달      |
| 저장 장치        | HDD, SSD, Flash 등          | 데이터 장기 저장          |
| 통신 장치        | UART, SPI, I2C, BLE, Wi-Fi | 장치 간 데이터 송수신       |
| 버스           | 데이터, 주소, 제어 버스             | 구성요소 간 데이터 전달 통로   |
| 타이머/카운터      | 내부 클럭, 비교기 등               | 시간 측정, 주기적 인터럽트 발생 |
| ADC/DAC      | 변환 회로                      | 아날로그 <-> 디지털 신호 변환 |
* ALU -> 실제 실행자 역할(연산)
	* Adder로 구성돼 있음(ASM -> ADD)
* CU -> Control Unit으로 제어 목적(데이터를 버스에 실어 옮김 등)
	* CU가 작업을 하려면 상태/정보를 어딘가에 기록할 필요가 있음
	* 레지스터를 사용함
	* 필요한 데이터/코드는 RAM/ROM/Cache 등에 저장됨 -> Stack, Data Segment, Heap 등

#### DMA(Direct Memory Access) Controller
* 메모리에 직접 접근
* CPU가 Memory R/W를 실행 -> PIO
* PIO 방식으로 대량의 데이터를 R/W하면 CPU가 다른 처리를 하기 어려움
![[Pasted image 20260423122623.png]]
![[Pasted image 20260423122640.png]]
![[Pasted image 20260423123510.png]]
-> 버스 충돌(bus contention)을 줄이기 위해 DMA Controller 사용
-> DMAC 설정 잘못되면 사고 발생, 주소, 사이즈 설정 유의

| 구분         | DMA                                         | Timer                 | RTC                   |
| ---------- | ------------------------------------------- | --------------------- | --------------------- |
| **역할**     | CPU 개입 없이 메모리 간 데이터 전송                      | 시간/주기 측정 및 인터럽트 발생    | 날짜/시간 유지<br>(배터리로 지속) |
| **제어 주체**  | 하드웨어/컨트롤러 자체 제어                             | 타이머 레지스터 설정           | 내장된 RTC 회로            |
| **CPU 사용** | 거의 없음(비동기 전송)                               | 일부 사용(인터럽트 기반)        | 없음 (저전력에서 독립 작동)      |
| **주요 기능**  | 고속 데이터 블록 이동                                | 딜레이 생성, PWM, 주기적 인터럽트 | 시스템 시계, 알람 기능         |
| **속도/성격**  | 빠름(병렬 수행 가능)                                | 빠름/정밀도 높음             | 느림(초 단위 기준)           |
| **인터럽트**   | O(전송 완료 시)                                  | O(시간 도달 시)            | O(알람 시)               |
| **예시**     | Memory <-> UART<br>Memory <-> ADC data send | 1s 간격 인터럽트, 모터 PWM 출력 | 날짜/시간 유지, 알람 설정       |
#### Timer
* 주변 장치 감시, 데이터 출력 등 시간 관련 처리를 위해 필수
* 카운터 레지스터를 통해 인터럽트 발생, CPU에 알림

#### RTC(Real Time Clock)
* 시간을 관리하기 위한 장치
* 내부 타이머가 동작하는 방식
* 시간 한 번 설정 시 전원이 켜 있는 동안은 시간 갱신

#### GPIO(General Purpose I/O)
* 외부 주변 장치로부터 인터럽트 신호에 사용하는 등, 범용 입출력에 사용

#### Register
* 작은 형태의 저장장치
* 스택 포인터, 링크 레지스터와 같이 특별한 의미로 사용되는 경우 존재
* CPU의 형태에 따라 다름
* 요즘 CPU는 16개의 레지스터 사용
	* PC(Program Counter) -> 명령어의 주소
	* Link Register -> 특정한 위치 점프 시 돌아오기 위한 주소 저장
	* Stack Pointer -> 스택에 데이터 I/O시 최상위 스택 주소 저장
	* 위 3개 전용 레지스터(High level Register)와 범용 레지스터 존재
	* 13, 14, 15 -> 전용 레지스터 번호
* 메모리 맵드 I/O, I/O 맵드 I/O
	* 메모리 맵드 : ROM, RAM과 공통 주소 이용, 특정 주소에 R/W를 수행함으로 주변장치 제어
		* ROM, RAM과 마찬가지로 주변장치의 레지스터도 메모리로 취급
	* 맵드 : ROM, RAM의 제어와 주변장치의 제어는 주소도, 사용하는 명령도 어

|           | Memory Mapped I/O                       | I/O Mapped I/O(isolatied I/O / Port-Mapped I/O) |
| --------- | --------------------------------------- | ----------------------------------------------- |
| 주소 공간     | 메모리 주소 공간과 공유                           | 별도 I/O 주소 공간 사용                                 |
| 명령어 사용    | 일반 메모리 접근 명령어 사용(MOV, LDR 등)            | 별도 명령어 사용(IN, OUT등)                             |
| 사용 예시     | RISC                                    | x86                                             |

#### 중앙 처리 장치
| 구성           | 요소                                           | 시험키워드                  | 비고                 |
| ------------ | -------------------------------------------- | ---------------------- | ------------------ |
| 프로그램 카운터(PC) | 다음에 실행할 명령어 주소 저장                            | 명령어 위치<br>순차 실행<br>분기  | PC값 -> 메모리 주소 지정   |
| 디코더(Decoder) | 명령어 해석, 제어 신호 생성                             | 명령 해석<br>제어 신호<br>오퍼코드 | 명령어 -> ALU/R 제어    |
| ALU          | 사칙/논리 연산 실행                                  | 사칙 연산, 논리 연산           |                    |
| 범용 레지스터      | 데이터 저장 및 연산 중간 값 보관                          | 임시 저장, 중간 값 보관, 연산     |                    |
| 시스템 레지스터     | 상태 정보, 제어 정보 저장(CPU 내부 설정용), 명령, 주소, 상태 레지스터 | 상태 제어, 플래그, 특수 목적      | PSW, SP, PC, FLAGS |
* 범용 -> 모든 용도로 사용 가능 => 데이터 저장/주소 저장
* 시스템 레지서터 -> 특별한 의미를 갖는 레지스터
	* 주소, 명령, CPU의 상태를 저장해놓는 상태 레지스터

#### CPU 명령 실행
* 명령 패치 사이클 -> 명령 디코드 사이클 -> 실행 사이클 -> 라이트 백 사이클
	* 패치 : Read | ROM으로부터 명령을 추출
	* 디코딩 : 해석 | 실행 준비
	* 실행 : 명령 실행
	* 라이트 백 : 명령 결과 외부로 출력
	* F-D-E-W 과정
		* 하나의 프로세스가 D 수행 시 뒤따르는 프로세스가 F를 수행 이후 동일
		* 병렬 프로그래밍 시 수행하며 
		* 해당 일련의 과정을 파이프라이닝이라 명칭함

#### 인터럽트
* 주변 장치로부터의 처리 요구를 CPU에 통지하기 위한 신호
* CPU가 인터럽트 신호를 받으면 인터럽트 벡터 테이블이라는 전용의 위치에 실행이 옮겨짐
* 인터럽트에는 CPU 내부의 타이머로부터 발생하는 타이머 인터럽트와 주변장치 등의 외부 HW로부터 발생하는 외부 인터럽트가 존재
* 데이터 교환 : 주변장치 <-> CPU / CPU <-> R => F, W 과정을 보인다
*
