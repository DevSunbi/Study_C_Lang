## 데이터 형식
#### 정수형 데이터 형식
* int, short, long, long long
* char(1), short(2), int(2, 4), long(4), long long(8)
* int는 시스템에 따라 크기가 다름 유의
#### 부동 소수점 데이터 형식 flaot, double
* 부동 소수점은 실수를 표현하는 데이터 형식
* 소수점 아래가 10자리, 앞자리가 3자리인 숫자와
* 소수점 아래 2자리, 앞자리 11자리 숫자를 동일한 크기로 표현하기 위해 소수점 위치를 고정(부동)하지 않고 필요에 따라 변경
* set position func를 사용해 정밀도 조정 가능
	* 근사치가 정확/부족해지는 효과 
#### L-value / R-value
* L : 수정 가능한 표현식, 메모리 위치를 가리키거나 수정 할 수 있는 표현식
	* 변수 이름, 배열 요소, 클래스 멤버
* R : 대입 연산자 우측에 나타남, 임시 값이나 메모리 상에 위치가 없는 값을 가리킴
	* 임시 객체, 리터럴
#### 형식 변환 : 명시적 형 변환
* float(), (float)()등 데이터 형식을 사용한 명시적 형 변환
* static_cast, const_cast, dynamic_cast, reinterpret_cast등 다양한 명시적 형 변환 지원
	* static_cast : 논리적 변경할 수 있는 형 변환 모두 가능
		* 상속 관계에 있는 포인터끼리 변환도 지원
	* const_cast :  포인터 및 레퍼런스 형식에서만 사용 가능
		* const, volatile 제거 할 때 사용
	* reinterpret_cast : 일반적 명시적 형 변환과 동일
		* const 사용하는 변환 대상은 사용 불가
	* dynamic_cast : 클래스의 포인터 및 레퍼런스 변수 간의 형 변환 시 사용
		* 안전한 다운캐스팅을 위해 사용
			* OOP에서 중요
			* 다형성을 사용하다 보면 부모 클래스의 포인터로 자식 객체 가리키는 경우 다수
			* 자식 클래스만의 고유 기능 사용할 때 dynamic_cast가 이 과정이 가능한지 확인해줌
				* 부모 클래스 타입의 포인터나 참조자를 통해 여러 자식 객체 관리(Upcasting)
				* 부모 포인터가 가리키고 있는 이 객체가 실제로 특정 자식 클래스가 맞는지 확인, 자식만의 메서드 호출 할 때 있음 -> 다운캐스팅
#### 비트 연산자
* XOR(^) => 하나만 1일 때 1로, 아니면 0으로
* AND(&)(Masking) => 동일 비트 비교, 모두 1일 때 1
* RightShift(>>)(\==\*2) : 비트 우측 이동, 최우측 비트 버려짐
* LeftShift(<<)(\==\/2) : 비트 좌측 이동, 최좌측 비트 버려짐

## 포인터와 메모리
#### 동적 메모리 할당
* 변수의 메모리의 크기를 동적으로 할당 할 수 있는 new. delete
	* datatype* valname = new datatype;
	* delete val_name;
	* 배열도 가능
		* datatype* valname = new datatype\[size]
		* delete\[] valname
#### 연산자 오버로딩
* (" ") -> 문자열 클래스의 객체로 해석 가능
* 산술 연산자는 원래 수를 더함 -> string을 붙히는데 사용 가능
* 특정한 클래스에서 접합 연산자로 변경해서 사용한다 재정의 -> operator overloading
#### 구조체
* 구조체 쓸 순 있는데 class가 더 좋음
* 구조체를 open class라고도 말 함
#### 정적 변수와 상수 변수
* 지역변수 : 해당 블록 내 효력
* 전역 변수 : 해당 파일 내 효력
* 자동지속 : 블록을 벗어나면 소명
* 정적 지속 : 프로그램 종료되면 소멸
* 정적 변수 : static
#### 레퍼런스
* 포인터 대신 사용 할 수 있는 변수 형식
* 일반 변수 : 값 저장
* 포인터 : 주소 저장
* 레퍼런스 : 변수에 또 다른 이름, 별칭 부여
	* 메모리 주소가 아닌 원본 변수 참조
	* 선언 시 변수 명 앞에 & 기호 사용
	* DataType& RefValName = ValName; 
	* 포인터의 위험성을 배제한 기능
## 예외 처리 구문
#### try, catch, throw
* 예외 발생 시 프로그램 제어 흐름 변경, 예외 처리
* try : 예외 발생할 수 있는 코드 블록 중괄호로 감쌈
* throw : 예외를 catch 블록으로 넘김
* catch : throw로 던진 예외를 받아 처리
* try {throw except } catch(ExceptType ExceptName) { }
#### catch
* 모든 예외 사항을 정의하기 어려움
* throw로 전달되는 모든 자료형에 대한 처리 필요
* else, default와 같이 catch(,,,)로 처리
	* 명시하지 않은 나머지 모든 예외 처리
#### 예외 전달 순서
* 예외 처리를 위해 사용한 throw에 대응하는 catch가 없을 때
	* throw와 같은 범위에서 catch로 우선 처리
	* catch가 없다면 상위 호출부에서 catch 처리 -> 스택 풀기
* 호출부에서 try catch로 감쌈 => 문제 없음
	* e.g ) fun1에 throw -> fun2에 throw -> main try catch => 이상 무
	* -> : 호출
	* 이 때 각 함수의 예외 발생 이후 모든 코드는 실행 되지 않는다
#### 어설션
* 코드를 검증, 예상치 못한 상황에서 프로그램 중단
	* 무결성 검증을 위한 매크로
	* 애지간해선 디버깅 사용함
## 예외 처리 생략과 실패 대응
#### 예외 처리 생략 - noexcept
* 함수가 예외를 처리하지 않음을 명시적으로 컴파일러에게 알리는 키워드
	* 예외처리를 위해 추가적인 컴파일로 작이 없어 컴파일 최적화도 용이
=> 잘 안씀
#### 예외 처리 실패 대응 - set_terminate
* catch가 없거나 형식에 맞지 않으면 프로그램 강제 종료
	* terminate라는 런타임 함수에서 프로그램 강제로 종료 시키는 abort 함수 호출
	* 강제 종료 전에 특정 동작 수행 가능
## 객체지향 이전의 프로그래밍 패러다임
#### 프로그래밍 패러다임
* 프로그램을 어떤 절차와 구조로 만들 것인지에 대한 스타일이나 접근 방법
* 언어가 지원하는 기능, 코드의 구조, 문제 해결 접근 방식 등에 다름
* 비구조적 프로그래밍, 절차적 프로그래밍 및 객체지향 프로그래밍 등
#### 비구조적 프로그래밍
* 코드를 구조화 하지 않고 작성
* 첫 번째 줄부터 마지막 줄까지 차례대로 실행
* 코드의 흐름을 이동하는 goto 문 사용이 특징
* 어셈블리어, 초창기 포트란
#### 절차적 프로그래밍
* 소스 코드를 여러 부분으로 나눠 활용, 프로시저를 이용해 구조화
	* 프로시저 : 일련의 코드 묶음, 함수를 지정
* C, 코볼, 포트란 등
* 코드를 논리 구조로 모듈화
	* 동일 기능 수행 코드 재작성 불필요
	* 재사용을 위해 기능을 묶어 라이브러리 모듈로 제공
* 구조화된 코드는 다른 사람이 쉽게 읽을 수 있음
```bash
/JMC /permissive- /ifcOutput "x64\Debug\" /GS /W3 /Zc:wchar_t /ZI /Gm- /Od /sdl /Fd"x64\Debug\vc145.pdb" /Zc:inline /fp:precise /D "_DEBUG" /D "_CONSOLE" /D "_UNICODE" /D "UNICODE" /errorReport:prompt /WX- /Zc:forScope /RTC1 /Gd /MDd /std:c++20 /FC /Fa"x64\Debug\" /EHsc /nologo /Fo"x64\Debug\" /Fp"x64\Debug\ConsoleApplication2.pch" /diagnostics:column 
```
* 컴파일러 옵션임
#### 절차적 프로그래밍 문제점
* 접근 제어 못하고 정보 공개
* 프로시저 간 논리 계층 구조 존재, 관련 코드 모두 확인 전 캐치 어려움
	* 프로그램의 기능을 프로시저로 분할하는 절차적 프로그래밍
	* top-down approach => 상위-하위 접근(큰-작은) 하향식 방식
* 지역 변수가 아닌 전역 변수를 변경
* 하위 프로시저를 직접 호출
#### 객체지향 프로그래밍 개념
* 논리적 단위 객체의 조합으로 프로그램을 구성
	* 작은 - 큰, 하위 - 상위 접근 상향식 방식
* 객체 지향 프로그래밍은 데이터(속성)와 함수(기능)을 가진 객체(하위)를 선정의
* 객체의 조합으로 프로그램 구성
	* 재활용 관점에서 유용 : 하위 구조의 활용도를 높이기 위해 범용성 우선 고려
#### 객체지향 프로그래밍 필요성
* 논리적 계층 관계 표현
	* 복잡하고 다양한 객체 간의 계층 관계 표현 가능
* 현실 모델링
	* 현실 세계의 사무링 가진 특징과 관계를 표현
* 접근 제어
	* 불필요한 정보 노출과 접근을 효과적으로 제어
* 추상화
	* 현실 세계의 대상을 논리적으로 구조화를 위해 단순화, 일반화 하는 과정
	* 구체적인 내용은 모름
